Elasticsearch

What is Elasticsearch?

Elasticsearch is an open-source, distributed search and analytics engine built on top of Apache Lucene.

It stores data as JSON documents and exposes everything through a RESTful HTTP API — no special driver, any language works.

Documents become searchable within ~1 second of indexing — making it suitable for real-time monitoring and live search.

💡 Analogy

Think of ES as the index at the back of a textbook — but for terabytes of JSON, with millisecond lookup. Your app's database is the full text; ES is the index that makes it searchable.

Common use cases

🔎 Full-text search

Product search, documentation portals, e-commerce — anywhere users type freeform queries.

📊 Log & metrics analytics

SIEM, APM, infrastructure monitoring — ingest high-volume event streams, query in real time.

📈 Business analytics

BI dashboards via Kibana — aggregations on large datasets without a data warehouse.

General Performance05
Elasticsearch

Built on Apache Lucene

Apache Lucene is the Java search library powering ES under the hood. It handles the inverted index, tokenization, and relevance scoring.

Elasticsearch adds on top of Lucene:

1

Distribution

Coordinates multiple Lucene instances across nodes into a single cluster.

2

REST API

Every operation — index, search, delete — is an HTTP call with a JSON body.

3

Operations

Cluster health, rebalancing, snapshots, index lifecycle management.

Inverted index — the core data structure

An inverted index maps each term → list of documents containing it. This is why full-text search is fast: ES looks up matching term postings instead of scanning every document. Actual cost depends on term frequency, scoring, filters, and result size.

💡 Analogy

A regular database index is like a library catalog sorted by title. An inverted index is like the index at the back of a book — sorted by keyword, pointing to where each word appears.

Architecture

Cluster & Nodes

🌐 Cluster

One or more nodes sharing the same cluster.name. They coordinate to store and serve data. You can talk to any node — it routes internally.

💻 Master-eligible node

Manages cluster state: which indices exist, shard placement, node membership. One is elected master. Should be dedicated in production (3 minimum to avoid split-brain).

💾 Data node

Stores shards and executes search & indexing. CPU and I/O intensive. Scale out these nodes as data grows.

🔀 Coordinating node

Receives client requests, fans out to relevant shards, merges results. Any node can coordinate. Can be dedicated for heavy query loads.

💡 Analogy — a city library network

The cluster = the monitoring platform. Each node = a server in the platform. The master = the coordinator that tracks where each shard lives. Data nodes = the machines holding log data.

🚧 Dev vs Production

A single node is a valid cluster — fine for development. In production: minimum 3 master-eligible nodes to prevent split-brain scenarios.

General Performance11
Architecture

Index

An index is a logical namespace for related documents — equivalent to a table in a relational DB, but more flexible.

Examples: app-logs-2026-06-03, web-logs-2026-06-03, security-events. You can search across multiple indices with a wildcard: *-logs-*.

Index settings (set at creation)

  • number_of_shards — how many primary shards (fixed after creation)
  • number_of_replicas — copies per shard (can change later)
  • analysis — custom analyzers
⚠️ Important

The number of primary shards is fixed at creation time. Changing it requires reindexing all data. Plan ahead.

Create an index

PUT /app-logs-000001 { "settings": { "number_of_shards": 3, "number_of_replicas": 1 }, "mappings": { "properties": { "@timestamp": { "type": "date" }, "service": { "type": "keyword" }, "level": { "type": "keyword" }, "message": { "type": "text" }, "latency_ms": { "type": "integer" }, "status_code": { "type": "integer" } } } }

Indexes & Aliases

An alias is a logical name that points to one or more indices. Applications query the alias, while Elasticsearch can rotate the concrete backing indices behind it.

Why aliases matter for logs

  • Keep the app writing to app-logs-write
  • Search across historical data with app-logs
  • Rotate daily or by size without changing app code
Trainer demo

Create app-logs-000001, attach aliases, index one document through app-logs-write, then search through app-logs.

PUT /app-logs-000001 { "aliases": { "app-logs": {}, "app-logs-write": { "is_write_index": true } }, "mappings": { "properties": { "@timestamp": { "type": "date" }, "service": { "type": "keyword" }, "message": { "type": "text" } }} } POST /app-logs-write/_doc { "service": "checkout", "message": "Payment gateway timeout after 2400ms" } GET /app-logs/_search

Architecture

Shards

A shard is a single Lucene instance — the actual unit of storage. An index is split into N shards distributed across nodes.

Documents are routed to a shard by:
shard = hash(_id) % number_of_shards

Why sharding?

  • Scale storage — data too large for one machine
  • Parallel queries — each shard searched concurrently
  • Horizontal scale — add nodes, ES rebalances shards
💡 Analogy

Shards are like separate partitions of the same log stream. When you search, ES asks each partition in parallel and merges the results.

Shard layout — 3 primaries, 1 replica, 2 nodes

NODE 1
P0 P1 R2
NODE 2
P2 R0 R1
P Primary shard  ·  R Replica shard

Sizing guideline

Aim for 10–50 GB per shard. Too many small shards waste overhead; too few large shards limit parallelism. Start with 1–3 primaries for most use cases.

General Performance14
Architecture

Replicas

A replica shard is a copy of a primary shard, always placed on a different node.

✅ Fault tolerance

If a node fails and its primary shards are lost, ES promotes the replica on another node to primary. No data loss.

⚡ Read throughput

Search requests can be served by either the primary or any replica. More replicas = more read capacity. They do add storage and replication work, but ES handles the coordination automatically.

💡 Analogy

Replicas are like photocopied chapters stored in a different room. If the original shelf burns down, you still have the copy — and normally both copies can be read simultaneously.

⚠️ Replica on a single node

With only 1 node, replicas stay UNASSIGNED — ES refuses to put a replica on the same node as its primary. This is normal in dev; cluster shows yellow status (data safe, not fully redundant).

# Check cluster health GET /_cluster/health # Change replica count on existing index PUT /app-logs-000001/_settings { "number_of_replicas": 2 }

Data Modeling

JSON Documents

Every piece of data in ES is a JSON document. A document is a flat or nested collection of field-value pairs.

Each document has a system metadata field _id (auto-generated or user-supplied), and lives inside an index.

Document anatomy

  • _index — which index it belongs to
  • _id — unique identifier within the index
  • _source — the original JSON you indexed
  • _score — relevance score (on search results)

Example document

{ "_index": "app-logs-000001", "_id": "evt-9f2a", "_source": { "@timestamp": "2026-06-03T10:15:30Z", "service": "checkout", "level": "WARN", "message": "Payment gateway timeout after 2400ms", "latency_ms": 2400, "status_code": 504, "trace_id": "trc-8842" } }
General Performance17
Data Modeling

Dynamic Mapping

When you index a document without defining a mapping, ES automatically infers field types from the first document it sees.

Auto-detection rules

  • "hello" → text + keyword
  • 42 → long
  • 3.14 → float
  • true → boolean
  • "2024-01-15" → date
  • {…} → object

⚠️ The trap — type conflicts

If the first document has "latency_ms": "fast", ES maps latency_ms as text. Subsequent documents with "latency_ms": 2400 will fail to index. Dynamic mapping is convenient for prototyping — risky in production.

Protect yourself

  • Set "dynamic": "strict" to reject unknown fields
  • Or "dynamic": false to silently ignore them
  • Define explicit mappings for production indices
# Check auto-generated mapping GET /app-logs-000001/_mapping

Data Modeling

Field Types

FIELD TYPE ANALYZED? USE FOR
text Yes ✓ Full-text search (titles, descriptions, body text)
keyword No ✗ Exact match, aggregations, sort, filter (IDs, tags, status, email)
integer / float No ✗ Numbers, ranges, aggregations (latency_ms, status_code, retries)
date No ✗ Timestamps, date ranges, date histograms (ISO 8601 or epoch ms)
boolean No ✗ Flags (in_stock, is_active, published)
nested No ✗ Arrays of objects where each object must be queried independently

⚠️ text vs keyword — the most common mistake

text is analyzed — "Hello World" becomes tokens ["hello", "world"]. You can't sort or aggregate on it. keyword stores the raw value. For fields you need both (search + aggregate), use fields: { raw: { type: "keyword" } }.

Data Modeling

Log Mapping Blueprint

A good log mapping separates fields used for filtering and aggregation from fields used for full-text search.

Structured fields

  • @timestamp as date
  • service, level, env as keyword
  • latency_ms, status_code as numbers
  • client_ip as ip, location as geo_point

Search fields

message should be text for full-text search, with a keyword subfield only when exact filtering is useful.

PUT /app-logs-000001 { "mappings": { "dynamic": "strict", "properties": { "@timestamp": { "type": "date" }, "service": { "type": "keyword" }, "env": { "type": "keyword" }, "level": { "type": "keyword" }, "message": { "type": "text", "fields": { "raw": { "type": "keyword", "ignore_above": 256 }} }, "latency_ms": { "type": "integer" }, "status_code": { "type": "short" }, "trace_id": { "type": "keyword" }, "client_ip": { "type": "ip" }, "location": { "type": "geo_point" } }} }

Data Modeling

Text Analysis & Analyzers

When you index a text field, ES runs it through an analyzer pipeline before storing it in the inverted index.

1

Character filter

Strip HTML, replace characters. "<b>Hello</b>" → "Hello"

2

Tokenizer

Split on whitespace/punctuation. "Hello World" → ["Hello", "World"]

3

Token filters

Lowercase, stemming, stopwords. ["Hello","World"] → ["hello","world"]

💡 Same analysis at query time

The same analyzer runs on your search query. So a query for "Running" matches documents with "run" because both get stemmed. Mismatch between index and query analyzers = missed results.

Built-in analyzers

standard

Default. Unicode tokenization + lowercase. Good for most western text.

english / spanish / arabic…

Language-specific stemming + stopwords. "running" → "run". Use when content language is known.

keyword

No analysis — entire input as one token (same as keyword field type).

# Test an analyzer interactively GET /_analyze { "analyzer": "english", "text": "The black cats are searching" }

Data Modeling

Nested Documents

An object field stores sub-documents inline. An array of objects is flattened by ES — this breaks cross-field queries.

⚠️ The object flattening trap

With type object, a query for {name: "cpu", status: "critical"} can match a doc where CPU is OK and disk is critical — because field values are merged across array items.

✅ Solution: nested type

Use "type": "nested" to store each array item as a hidden sub-document. ES then preserves the relationship between fields within each object.

# Mapping with nested type PUT /system-events/_mapping { "properties": { "message": { "type": "text" }, "checks": { "type": "nested", "properties": { "name": { "type": "keyword" }, "status": { "type": "keyword" } } } } } # Query nested documents correctly { "nested": { "path": "checks", "query": { "bool": { "must": [ { "term": { "checks.name": "cpu" }}, { "term": { "checks.status": "critical" }} ]}} }}

Query DSL

Query Context vs Filter Context

Query context — "how well does it match?"

Computes a relevance score (_score) for each matching document using BM25. Use for full-text search where ranking matters.

"query": { "match": { "title": "elasticsearch guide" } } → returns documents scored by relevance

Filter context — "does it match? yes/no"

No score computed. Cached by ES — much faster for structured data. Use for exact values, ranges, boolean flags.

"filter": { "term": { "status": "published" } } → yes/no, cached, no scoring overhead
Best practice

Use filter for structured log data (service, level, status code, time range) and query only for full-text fields like message. Filters are cached and significantly faster.

General Performance27
Query DSL

Common Leaf Queries

match — full-text search

Analyzes the query string and searches a text field. The standard full-text query.

{ "match": { "message": "payment timeout" }}

term — exact value

No analysis — exact match. Use on keyword, boolean, integer fields.

{ "term": { "level": "ERROR" }}

terms — multiple values

Like SQL IN (…).

{ "terms": { "service": ["checkout", "payments"] }}

range — numeric or date range

Operators: gte, gt, lte, lt.

{ "range": { "latency_ms": { "gte": 1000 }}}

match_all — return everything

All documents, score = 1.0. Useful when only filtering.

{ "match_all": {} }

multi_match — search across fields

Same as match but on multiple fields.

{ "multi_match": { "query": "cache warmed", "fields": ["message", "error_detail"] }}
General Performance28
Query DSL

The Bool Query

The bool query is the Swiss army knife — it combines other queries using logical clauses.

must

Must match. Contributes to score. Equivalent to AND.

filter

Must match. No score, cached. Equivalent to AND (faster).

should

Optional. Boosts score if it matches. Equivalent to OR. With minimum_should_match: required count.

must_not

Must NOT match. No score, cached. Equivalent to NOT AND.

Example — log triage

GET /app-logs-*/_search { "query": { "bool": { "must": [ { "match": { "message": "payment timeout" }} ], "filter": [ { "term": { "service": "checkout" }}, { "range": { "latency_ms": { "gte": 1000 }}} ], "must_not": [ { "term": { "level": "DEBUG" }} ] }} }
Rule of thumb

Put full-text queries in must, structured filters in filter. The filter clause will be cached and skip scoring — significantly faster for high-traffic endpoints.

General Performance29
Query DSL

Aggregations

Aggregations compute analytics over matching documents. Think SQL GROUP BY + aggregate functions — distributed across shards and fast when mappings, shard sizing, and hardware are right.

Bucket aggregations — group by

  • terms — group by field value
  • date_histogram — group by time interval
  • range — group by numeric ranges

Metric aggregations — compute stats

  • avg, min, max, sum
  • stats — all metrics in one
  • cardinality — distinct count (HyperLogLog, approximate)
💡 Analogy

Aggregations are like a pivot table in Excel — but distributed across the cluster, so very large datasets can be summarized without loading everything into your application.

Agg — avg latency per service

GET /app-logs-*/_search { "size": 0, // no doc hits, aggs only "aggs": { "by_service": { "terms": { "field": "service", "size": 10 }, "aggs": { "avg_latency": { "avg": { "field": "latency_ms" } } } } } }

Grok — Parsing Unstructured Logs

Grok maps named regex patterns to log lines, turning raw strings into structured fields.

Syntax: %{PATTERN:field}

Matches the named pattern and assigns the captured value to the given field name.

# Raw line: 192.168.1.1 - - [15/Jan/2024:10:00:00 +0000] "GET /api HTTP/1.1" 200 512 # Use built-in pattern: %{COMBINEDAPACHELOG} # Produces fields: # clientip, timestamp, verb, request, response, bytes

📚 Useful built-in patterns

  • IP, HOSTNAME, URI
  • NUMBER, INT, POSINT
  • HTTPDATE, TIMESTAMP_ISO8601
  • COMBINEDAPACHELOG, NGINXACCESS

Applications

Elasticsearch in Application Code

Applications usually treat Elasticsearch as a search and analytics read model. The primary database remains the source of truth; Elasticsearch is updated asynchronously or through controlled bulk jobs.

Integration patterns

  • REST calls from any language for simple operations
  • Official clients for retries, connection pooling, and typed APIs
  • Bulk indexing for backfills and batch ingestion
  • Aliases so app code does not change when indices rotate

Production cautions

Handle retries carefully, do not index invalid mappings blindly, and keep search requests bounded with time filters, pagination, and explicit result sizes.

Same REST idea, different clients

# REST curl -XGET "localhost:9200/app-logs-*/_search" \ -H "Content-Type: application/json" \ -d '{"query":{"match":{"message":"timeout"}}}' # Python client es.search(index="app-logs-*", query={"match": {"message": "timeout"}}) # Java client, same operation conceptually client.search(s -> s.index("app-logs-*") .query(q -> q.match(m -> m.field("message").query("timeout"))), LogEvent.class);
